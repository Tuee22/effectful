Therac-25: Where the Proof Boundary Killed

Matt uses the Therac-25 radiation therapy disaster to make the proof boundary tangible. This analysis traces how the case study clarifies — and complicates — the concept.

Historical Computer Science Ethical Mathematical Cross-cutting

The Three Machines

Matt tells the Therac story in three phases during the conversation. Each phase represents a distinct configuration of the proof boundary — and a different relationship between human judgment, software, and physical safety.

Phase A — The Analog Machine (Pre-software)

100% analog controls — “a bajillion buttons and dials.” Required specialized radiation technicians (not nurses, not radiologists). Extremely reliable 1930s-era technology. The human was fully in the loop: turning dials directly controlled the radiation emitter. No software, no abstraction, no formal language between intent and output.

The proof boundary: human judgment → analog machine → radiation. There was nothing to prove — the system was transparent. What you dialed is what you got.

Phase B — The Therac-20 (Software with hardware safety)

Introduced software control — nurses could enter instructions via a terminal rather than manipulating physical dials. But the machine retained a mechanical barrier that capped radiation dose at approximately 100 units regardless of what the software commanded.

The software had the same race condition bug that would later kill patients in the Therac-25. But the hardware safety caught the error. Patients received overdoses, but survived — the mechanical barrier absorbed the excess energy.

The proof boundary moved here: human judgment was partially replaced by software. A formal language now stood between human intent and radiation delivery.

Phase C — The Therac-25 (Software without hardware safety)

Removed the mechanical barrier — trusted software alone to control radiation dosage. The same race condition remained: an operator entering “2 units… correction, 5 units” could cause the machine to deliver 5,000 units. Patients died. No formal verification was done on the software.

The proof boundary was in the same position as Phase B, but verification was reduced. The safety net was removed without adding any replacement verification.

What Matt Is Really Saying

The Critical Move

“The most interesting part of the story is going from A to B and that’s actually where the proof boundary got moved.”

The move from Phase A to Phase B was the critical moment — when a formal language (a programming language) was inserted between human intent and radiation delivery. This is what created the opportunity (and obligation) to do proofs.

Matt’s key claim:

Matt: “The proof boundary moving actually consisted of was the creation of a language.”

This is profound: the proof boundary moves when someone creates or selects a formal representation of human judgment. The language is the boundary. Before software, there was no formal system to verify. After software, there was a formal system that could be verified — and therefore should be verified.

The move from B to C (removing the hardware safety) was not a move of the proof boundary itself. It was a reduction in the verification done at an already-moved boundary. This distinction is subtle but essential: the boundary’s position didn’t change, but the safety margin around it was eliminated.

The Verifiable vs Verified Distinction

One of the sharpest exchanges in the entire conversation. Keith and Matt zero in on the precise difference between what could be checked and what was checked:

Keith: “What was verifiable was different between B and C.”

Matt: “What was verifiable was the same. What was verified was different.”

Verifiable? Yes — could have done formal proofs Yes — could have done formal proofs
Verified? Partially (hardware safety as backup) No (hardware safety removed)

The proof boundary was in the same place for both machines. What differed was whether verification was actually performed. This is Matt’s distinction between the boundary existing and the proofs being done. The boundary is a fact about where automation begins. Proofs are the obligation that the boundary creates.

Mathematical Dimension: The Race Condition

The specific bug was a race condition — a timing-dependent error in concurrent state management:

  1. Operator enters dose (e.g., 2 units for electron mode)
  2. Operator quickly corrects to a different mode with 5 units
  3. Software doesn’t properly reset internal state during the mode switch
  4. Machine delivers the energy level of the first mode with the dose setting of the second mode — a massive overdose

This type of bug occupies a special category in computer science:

  • Undetectable by code review — it depends on timing, not logic that is visible in a single code path
  • Undetectable by standard testing — it only occurs with specific input sequences entered at specific speeds
  • Detectable by formal verification — tools like TLA+ can model all possible interleavings of concurrent state transitions and check invariants against every reachable state

This is why Matt argues formal proofs are essential at the proof boundary: human code review and testing cannot reliably catch these bugs. The state space is too large for human minds to enumerate, and too sparse for random testing to cover. Only exhaustive formal exploration of the state space can provide assurance.

Race Condition (formal)

A race condition occurs when the correctness of a program depends on the relative timing or ordering of events (e.g., thread scheduling, interrupt timing, user input speed) that are not controlled by the program’s logic. Formally, it means there exist two or more interleavings of operations that produce different results, at least one of which violates a system invariant.

Ethical Dimension: Layers of Responsibility

The Therac-25 disaster implicates multiple actors across multiple layers of decision-making. Each had the power — and arguably the obligation — to insist on formal verification at the proof boundary:

  1. The software developer — Wrote buggy code, did not use formal verification, did not model concurrent state transitions
  2. The hardware engineer (Therac-25) — Removed the mechanical safety barrier that protected patients in the Therac-20, trusting software alone
  3. The manufacturer (AECL) — Sold the product, claimed it was safe, responded slowly to incident reports
  4. The regulator — Certified the device without requiring formal verification of safety-critical software
  5. The hospital administrator — Purchased and deployed the machine, chose to automate radiation therapy decisions
  6. The insurance company — Priced the risk of the device (poorly), creating economic incentives that did not reflect actual danger

Matt argues the hospital administrator is the key decision-maker for the proof boundary — they chose to automate, to replace the specialized radiation technician with a nurse using software. But the moral error was shared across all levels. Everyone who could have insisted on formal proofs and didn’t bears a portion of responsibility for the deaths.

The Accountability Gap

No single actor bore full responsibility. The software developer didn’t decide to remove the hardware safety. The hardware engineer didn’t write the buggy software. The hospital administrator didn’t know about the race condition. The regulator didn’t have formal verification standards for software. Each actor’s local decision was defensible; the systemic failure was catastrophic. This is the proof boundary’s ethical challenge: who is responsible when the obligation to verify falls through every crack?

Computer Science Dimension: What Formal Verification Would Have Caught

If TLA+ or a similar formal verification tool had been applied to the Therac-25 software:

  1. The state space model would enumerate all possible sequences of operator inputs, including rapid corrections and mode switches
  2. The race condition would appear as a reachable state where dose > safe_limit
  3. The model checker would flag this as an invariant violation: “there exists a sequence of valid operator actions that leads to an unsafe dose”
  4. The fix would be required before deployment — the invariant violation would be a blocking finding
Matt: “They did proofs on the next generation of software and no more patient stuff.”

This is the strongest evidence for Matt’s thesis: formal proofs worked when they were finally applied. The proof boundary didn’t need to move back — it just needed the proofs to be done. The problem was never automation itself. The problem was automation without verification.

The Proof Worked

After formal methods were applied to subsequent radiation therapy software, the class of errors that killed Therac-25 patients was eliminated. This is not a theoretical claim — it is a historical fact. The proof boundary stayed in the same position (software still controlled the radiation), but the proofs were actually performed. The outcome changed from catastrophe to safety.

Cross-Reference: The Book’s Treatment

The book (in its catastrophes chapter) covers the Therac-25 alongside several other cases:

  • The Pentium FDIV bug (Intel, 1994) — a hardware division error that produced incorrect floating-point results
  • The Ariane 5 explosion (ESA, 1996) — a software type conversion overflow that destroyed a $370 million rocket
  • Toyota’s unintended acceleration (2009–2011) — software defects in throttle control linked to multiple deaths

Each case follows the same four-step pattern:

  1. Human judgment replaced by software
  2. Software not formally verified
  3. Bug causes catastrophe
  4. Formal methods applied afterward → problem solved

This pattern is the proof boundary thesis in four steps. The consistency across wildly different domains (medical devices, microprocessors, rockets, automobiles) is what gives Matt’s argument its force. It is not a one-off cautionary tale. It is a recurring structural failure that has the same cause and the same fix every time.

Keith’s Learning Edge

By the end of the Therac discussion, Keith demonstrates real understanding of the distinction between the boundary’s position and the verification performed at it:

Keith: “The proof boundary itself was the same… how the system behaved and what was verifiable was different between B and C but the proof boundary remained the same.”

He has grasped that the boundary’s position is independent of whether proofs are done. The boundary moved at A→B (when software was introduced). It stayed in the same place at B→C (when the hardware safety was removed). What changed between B and C was not where the boundary was, but how much faith was placed in unverified software.

This is genuine progress from the confusion earlier in the conversation, where Keith conflated the boundary’s existence with the act of verification. The Therac story gave him a concrete example that separated the two concepts: the boundary is where automation begins, and proofs are what you do there. They are related but distinct.

Why the Therac Story Works

The Therac-25 succeeds as a pedagogical tool because it provides three data points (A, B, C) that let the listener triangulate the concept. With only two phases, the boundary’s position and the verification status would be confounded — you couldn’t tell which one mattered. Three phases let you hold one variable constant while the other changes. Phase B and C share the same boundary position but differ in verification. Phase A and B differ in boundary position. This is why Keith finally gets it: the three-phase structure is a natural experiment.
