Valid vs Sound: The Gap Proofs Can't Close

A technical distinction from formal logic that is central to understanding the limits of the proof boundary — and why Keith keeps needing to be reminded.

Mathematical Conceptual Computer Science Psychological

The Distinction

Valid Proof

Given these input assumptions (axioms/premises), we can logically deduce this output statement. A proof checker verifies validity. “If A then B” — the logic is correct.

Sound Proof

A valid proof where the input assumptions are also true in the real world. Soundness requires correspondence between the formal model and reality.

Matt: “A valid proof is like a proven theorem, right? It’s saying given these input assumptions we can logically deduce this output statement. That’s a valid proof, and a proof checker checks that the proof is valid… what it can’t do is verify that a valid proof is sound, which is to say that the input assumptions actually are correct in the real world. Because the proof checker and software in general has no way of comparing the data it’s been given about the real world to the actual real world. It just has to believe it. So garbage in, garbage out.”

Why This Matters for the Proof Boundary

Valid Proof Sound Proof
What it checks Internal logical consistency Correspondence to reality
Who/what can verify Machine (proof checker) Humans (judgment)
Decidable? Yes (for well-formed proofs) No (requires real-world knowledge)
Example “If max_dose=100, then dose<=100” “Is 100 the right max dose for this patient?”

The proof boundary sits exactly at the gap between valid and sound. On one side: what machines can verify (validity). On the other: what requires human judgment (soundness — are the assumptions correct?).

This is arguably the most precise way to define the proof boundary: it is the line between validity-checkable and soundness-requiring claims.

The Garbage-In-Garbage-Out Principle

Matt reduces it to: “garbage in, garbage out.” A valid proof with wrong premises is perfectly logical nonsense.

  • Premise: “Patients can safely receive up to 10,000 units.” Conclusion: “5,000 units is safe.” Valid? Yes. Sound? Deadly wrong.
  • Premise: “This LLM output is a correct legal citation.” Conclusion: “We can submit this to court.” Valid? Yes. Sound? See Mata v. Avianca.

Psychological Dimension: Why Keith Forgets

Keith: “Two words again — I know you explained them before but remind me.”

This forgetting pattern reveals something important:

  1. The terms are counterintuitive. In everyday English, “valid” and “sound” are near-synonyms. The technical distinction overloads familiar words with precise meanings that fight everyday intuition.
  2. The distinction is uncomfortable. It means that even perfect formal verification has a gap — the premises might be wrong. For someone building systems, this is anxiety-inducing. It is easier to forget the gap than to live with it.
  3. Keith’s practical orientation. As a software developer, Keith thinks in terms of “does it work?” — a question that conflates validity and soundness. Separating them requires shifting from pragmatic to philosophical thinking.

Mathematical Dimension: Godel's Shadow

This gap between valid and sound is related to (but distinct from) Godel’s incompleteness theorems. Godel showed that any sufficiently powerful formal system contains true statements that cannot be proven within the system. The valid/sound gap shows that even provable statements might not correspond to reality.

Together, these results establish permanent limits:

  • Godel: Not everything true can be proven.
  • Valid/Sound gap: Not everything proven is true (about the real world).
  • Turing’s Halting Problem: Not everything can be decided algorithmically.

These three limits form the mathematical foundation for why the proof boundary can never be fully eliminated — there will always be a domain that requires human judgment.

For deeper treatment, see the Godel, Russell, and Turing chapters in the book.

Computer Science Dimension

In software engineering, this distinction manifests as:

  • Type systems check validity: “this function returns an integer” — but not whether it returns the right integer.
  • Unit tests check specific cases of validity: “f(2) returns 4” — but not whether f should return x² or 2x for all inputs.
  • Formal verification (TLA+, Lean, Coq) checks validity against a specification — but the specification itself could be wrong.
  • The specification is where soundness lives — and specifications are written by humans.

Each layer adds confidence, but none closes the valid/sound gap completely.

Ethical Dimension

The valid/sound gap has moral implications: who is responsible when a valid-but-unsound system causes harm?

  • The Therac-25 software could have been formally verified (validity) — but the specification of safe dose ranges is a human judgment (soundness).
  • If the specification was wrong, the proof would have been valid but the patients would still die.
  • This means formal proofs do not absolve humans of moral responsibility — they just change where the responsibility lies (from code correctness to specification correctness).

Keith's Key Insight (Even If He Doesn't Know It Yet)

The Deepest Implication

When Keith says “no proof can validate that its input assumptions are correct” — he has actually grasped the deepest implication of the proof boundary. The boundary exists precisely because there will always be assumptions that require human judgment.

This is not a bug in the concept. It is the entire point.
